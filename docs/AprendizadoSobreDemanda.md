# Aprendizado Sobre Demanda

Conceitos agnósticos a linguagem, framework e ORM — valem independente do que você usar no futuro. Cada um aparece diretamente nesse projeto.

---

## 1. Programação Assíncrona e I/O não bloqueante

**Por que aparece aqui:** toda a camada de banco e boa parte dos testes rodam em modo async. O script de ingestão vai fazer múltiplas chamadas HTTP ao GitHub e ao LLM — entender quando e por que usar async evita que você bloqueie o processo inteiro esperando uma resposta de rede.

**O que entender:**

- O que é um event loop e por que ele existe
- Diferença entre I/O-bound e CPU-bound — e por que async só ajuda no primeiro
- O que significa "não bloqueante": a thread não fica parada esperando, ela passa para outra tarefa
- Diferença entre concorrência (fazer várias coisas alternando) e paralelismo (fazer ao mesmo tempo)
- Quando async complica mais do que ajuda (operações puramente computacionais, scripts simples)

## 2. Migrações de banco de dados

**Por que aparece aqui:** o banco ainda não tem nenhuma migration criada. Sem isso, as tabelas não existem. Além disso, toda mudança de schema no futuro (adicionar coluna, renomear campo) precisa passar por uma migration — fazer `ALTER TABLE` manual em produção é uma das formas mais rápidas de quebrar tudo.

**O que entender:**

- O que é versionar schema como código e por que isso importa
- O ciclo `upgrade` / `downgrade`: toda mudança deve ser reversível
- Idempotência: rodar a migration duas vezes não pode quebrar o banco
- A diferença entre migration autogenerated (o ORM detecta as mudanças) e escrita à mão — e quando cada uma falha
- Por que migrations nunca devem ser editadas depois de aplicadas em produção

## 3. Injeção de Dependência

**Por que aparece aqui:** toda a API usa DI para receber a sessão de banco e a validação de API Key. Sem entender o padrão, adicionar qualquer endpoint novo vai parecer mágica — e você vai copiar código sem saber o que está fazendo.

**O que entender:**

- O problema que DI resolve: desacoplar "quem usa" de "quem cria"
- Como um container/framework de DI resolve dependências automaticamente em runtime
- Por que DI facilita testes: você substitui a dependência real por uma fake sem mudar o código que a usa
- A diferença entre injetar por construtor, por parâmetro de função e por decorador — são formas diferentes de expressar o mesmo conceito

## 4. Design de APIs REST

**Por que aparece aqui:** você vai consumir esse backend num site e potencialmente expor para outros. Entender as convenções evita que você precise reescrever contratos depois.

**O que entender:**

- Semântica dos verbos HTTP: `GET` é idempotente e não tem efeito colateral, `POST` cria, `PUT`/`PATCH` atualizam
- Semântica dos status codes: a diferença real entre 400, 401, 403, 404, 422 e 500
- Estrutura de URLs: recursos como substantivos (`/repositories/123`), não verbos (`/getRepository`)
- Paginação: offset vs cursor — quando cada um escala melhor e por que cursor é mais robusto
- Versionamento de API (`/v1/`) e por que existe

## 5. Autenticação e Autorização (conceitos)

**Por que aparece aqui:** você tomou a decisão de usar API Key em vez de JWT. Vai tomar decisões parecidas em todo projeto. Entender os modelos evita escolher errado.

**O que entender:**

- Autenticação ("quem é você?") vs Autorização ("o que você pode fazer?") — são problemas separados
- API Key: simples, estático, ideal para clientes conhecidos (um único site consumindo)
- JWT: stateless, carrega claims, ideal para múltiplos usuários com sessões
- OAuth2: delegação de acesso, ideal quando um serviço precisa agir em nome de um usuário
- Por que armazenar senhas em plaintext é catastrófico e o que hashing resolve (bcrypt, argon2)
- O que é um token de curta duração e por que tokens não expiram sozinhos sem revogação

## 6. Consumo de APIs externas: rate limiting, paginação e resiliência

**Por que aparece aqui:** o script de ingestão vai fazer dezenas (potencialmente centenas) de chamadas ao GitHub e ao LLM. APIs externas falham, limitam e paginate — lidar com isso de forma ingênua quebra a ingestão no meio.

**O que entender:**

- O que é rate limit e como interpretar os headers de resposta (`Retry-After`, `X-RateLimit-Remaining`)
- Estratégias de retry: quando tentar de novo faz sentido e quando não (erros 4xx geralmente não valem retry)
- Backoff exponencial: por que esperar cada vez mais entre tentativas, em vez de tentar em loop
- Paginação em APIs: offset/page vs cursor vs Link header — como saber qual a API usa
- Idempotência em chamadas externas: o que acontece se você chamar duas vezes e como evitar duplicações

## 7. Testes: unitários vs integração e o que mockar

**Por que aparece aqui:** o projeto já tem testes, mas o script de ingestão ainda não tem. Escrever o teste errado (ex: testar contra a API real do GitHub) torna a suíte lenta, não-determinística e cara.

**O que entender:**

- A diferença de propósito: unitário testa lógica isolada, integração testa a junção de partes reais
- O que é um mock/stub/fake e quando usar cada um — mockar o GitHub faz sentido, mockar a função de cálculo de score não
- Por que testes que dependem de rede ou tempo são frágeis
- O conceito de test fixture: estado pré-configurado para que o teste seja repetível
- Por que testes devem ser independentes entre si (um não pode depender do outro ter rodado antes)

## 8. Containerização: o que Docker resolve

**Por que aparece aqui:** o ambiente de desenvolvimento inteiro roda via Docker Compose. Sem entender o que está acontecendo, qualquer problema vira um mistério.

**O que entender:**

- O problema que containers resolvem: "funciona na minha máquina" — isolamento de ambiente
- Diferença entre imagem (template imutável) e container (instância rodando)
- O que é uma rede Docker e por que a API consegue falar com o banco usando o nome do serviço (`db`) em vez de `localhost`
- O que é um volume e por que sem ele os dados do Postgres somem ao reiniciar o container
- Docker Compose como orquestrador local: `depends_on`, `healthcheck`, `env_file`

## 9. Engenharia de Prompt (LLM)

**Por que aparece aqui:** o `LLMClient` já está implementado, mas o prompt atual é genérico. A qualidade do score de amigabilidade depende diretamente de quão bem o prompt instrui o modelo — e prompt mal escrito é o tipo de bug mais difícil de debugar.

**O que entender:**

- A diferença entre o papel `system` (instrução de comportamento) e `user` (entrada do usuário)
- O que é temperatura: valores baixos (~0.2) geram respostas mais determinísticas, valores altos (~0.8) mais criativas — para scoring, você quer baixo
- O que são tokens e como eles afetam custo e limite de contexto
- Técnicas básicas: few-shot (dar exemplos no prompt), chain-of-thought (pedir que o modelo raciocine antes de responder)
- Como validar a saída do LLM: modelos alucinam e podem retornar JSON malformado ou scores fora do intervalo esperado
